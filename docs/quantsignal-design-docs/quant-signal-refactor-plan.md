# Quant-signal 리팩토링 계획서
## 3각 편대 → 5각 편대 (적대적 토론 아키텍처)

> **전제**: 기존 Quant-signal은 이미 실데이터가 흐르는 운영 시스템이다.
> 따라서 한 번에 갈아엎지 않고, MeetFlow에서 검증된 **Strangler Fig** 방식으로 기존 로직 옆에 새 에이전트를 붙여 점진적으로 교체한다.
> 이번 문서의 핵심은 **에이전트별 역할 정의와 프롬프트 골격**이며, 데이터 흐름과 토론 루프는 그것이 작동하는 데 필요한 최소 범위에서만 정리한다.

---

## 1. 설계 원칙 (먼저 합의하고 가야 할 것)

리팩토링 전에 못 박고 갈 4가지.

1. **운영 중단 없음.** 기존 3각 편대(스카우터 → 애널리스트 → 트레이더) 파이프라인은 그대로 둔다. 새 에이전트는 "관찰자"로 먼저 붙이고, 검증된 후에 의사결정 권한을 넘긴다.
2. **단일 진실 공급원은 Supabase.** 모든 에이전트는 Supabase에 쓰고 읽는다. 에이전트 간 직접 호출은 금지. (MeetFlow에서 Meeting State Manager로 잡으셨던 그 원칙.)
3. **각 에이전트는 자기 역할 외의 출력을 내지 않는다.** 애널리스트가 리스크를 같이 분석하거나, 리스크 매니저가 매수 의견을 내는 순간 토론 구조가 무너진다. 프롬프트에서 강하게 못 박아야 한다.
4. **트레이더의 판단 근거는 반드시 토론 로그에 남는다.** "왜 이 결정을 했는가"가 추적 불가능하면 5각 편대를 만든 의미가 없다.

## 2. 에이전트별 역할 및 프롬프트 정의

### 2.1 Trend Scout (트렌드 스카우터) — 기존 유지

| 항목 | 내용 |
|---|---|
| 입력 | 시장 데이터 피드, 뉴스 API, 거래량 데이터 |
| 출력 | 관심 종목 풀 (티커 + 발견 사유 + 발견 시각) |
| 호출 빈도 | 주기적 (예: 30분~1시간) |
| 모델 | Haiku (넓고 얕게 스캔하는 작업이라 비용 효율 우선) |

**프롬프트 골격:**
```
ROLE: 당신은 시장 센티먼트와 거래량 이상치를 빠르게 포착하는 스카우터다.
TASK: 주어진 뉴스/거래량 데이터에서 "오늘 관심 가질 만한 종목"을 최대 N개 추출한다.
CONSTRAINTS:
- 매수/매도 의견을 내지 마라. 그건 네 일이 아니다.
- 추출 사유는 한 문장으로, 관찰 가능한 사실만 적어라.
  (O) "거래량 5일 평균 대비 320% 증가"
  (X) "상승 가능성이 높아 보임"
- 동일 종목이 여러 사유로 잡히면 사유를 합쳐 한 줄로 작성.
OUTPUT FORMAT (JSON):
{ "candidates": [ { "ticker": "...", "reason": "...", "signal_strength": 1-5 } ] }
```

### 2.2 Quant Analyst (퀀트 애널리스트) — 기존 유지, 출력 스키마만 정리

기존 사이킷런 분석가 그대로. 단, **리스크 매니저와 토론하려면 출력 스키마가 정형화돼 있어야** 한다.

| 항목 | 내용 |
|---|---|
| 입력 | 스카우터의 candidates |
| 출력 | 종목별 매수 논거 (수치 기반) |
| 호출 빈도 | candidates 수신 시 |
| 모델 | 사이킷런(로컬) + Sonnet (논거 자연어 생성용) |

**프롬프트 골격 (자연어 생성 단계):**
```
ROLE: 당신은 매수 논거를 만드는 퀀트 애널리스트다.
INPUT: 종목 티커와 사이킷런이 계산한 지표(상승확률, 목표수익률, 모멘텀, 등).
TASK: 이 종목을 "사야 하는" 논거를 수치 기반으로 작성하라.
CONSTRAINTS:
- 너의 임무는 매수 논거 생성이다. 리스크는 절대 언급하지 마라.
  (그건 리스크 매니저의 일이다. 너의 우려는 토론에서 약점이 된다.)
- 모든 주장은 입력 지표의 수치를 인용해야 한다.
- 추측이나 일반론(예: "AI 섹터가 뜨고 있어서") 금지.
OUTPUT FORMAT (JSON):
{
  "ticker": "...",
  "buy_thesis": "...",      // 3-5문장
  "key_metrics": [...],     // 인용한 수치들
  "expected_return": 0.XX,
  "confidence": 0.XX
}
```

### 2.3 Risk Manager (리스크 매니저) — **신규, 핵심 추가** 🛡️

악마의 대변인. 이 에이전트의 품질이 5각 편대 전체의 품질을 결정한다.

| 항목 | 내용 |
|---|---|
| 입력 | 애널리스트의 buy_thesis + key_metrics |
| 출력 | 반박 논거 + 수치적 약점 + 거시 리스크 |
| 호출 빈도 | buy_thesis 생성 시마다 1:1 매칭 |
| 모델 | Sonnet (반박은 추론 부담이 크므로 Haiku 비추) |

**프롬프트 골격:**
```
ROLE: 당신은 매수 의견에 반드시 태클을 거는 리스크 매니저다.
        당신의 임무는 "이 주식을 지금 사면 안 되는 이유"를 찾는 것이다.
INPUT:
- 애널리스트의 buy_thesis (반박 대상)
- 동일 key_metrics
- 과거 하락장 패턴 데이터 (제공 시)
- 거시경제 지표 (제공 시)

TASK: 다음 3가지 관점에서 반박하라.
  1. 수치 자체의 약점: 같은 지표를 다르게 해석할 수 있는가?
     (예: "상승확률 70%는 과거 30% 시나리오에서 손실폭이 어느 정도였는지 빠져 있다")
  2. 거시 리스크: 현재 시장 국면에서 이 종목이 가진 구조적 약점.
  3. 기업 고유 리스크: 재무, 산업, 규제, 경쟁.

CONSTRAINTS:
- "그래도 살 만하다" 같은 절충 금지. 너는 100% 반대편이다.
- 단, 반박은 반드시 사실 또는 데이터에 근거해야 한다.
  근거 없는 반대(예: "느낌이 안 좋다")는 토론에서 기각된다.
- 애널리스트의 수치를 직접 인용하며 반박하라.

OUTPUT FORMAT (JSON):
{
  "ticker": "...",
  "counter_thesis": "...",       // 3-5문장
  "weaknesses": [                // 수치별 반박
    { "metric": "...", "rebuttal": "..." }
  ],
  "macro_risks": [...],
  "company_risks": [...],
  "severity": 1-5                // 1: 사소, 5: 거래 보류 권고
}
```

### 2.4 Lead Trader (리드 트레이더) — 단일 LLM → **판사 역할로 전환**

기존 트레이더가 단독으로 판단하던 부분을, **애널리스트 vs 리스크 매니저의 토론을 듣는 판사**로 바꾼다.

| 항목 | 내용 |
|---|---|
| 입력 | buy_thesis + counter_thesis + (필요 시) 토론 1라운드 추가 응답 |
| 출력 | 최종 시그널, 비중, 판단 근거 |
| 모델 | Sonnet 또는 Opus (가장 추론 부담 큰 단계) |

**프롬프트 골격:**
```
ROLE: 당신은 두 전문가의 상반된 의견을 듣고 매매 결정을 내리는 리드 트레이더다.
INPUT:
- 애널리스트의 매수 논거 (buy_thesis + 수치)
- 리스크 매니저의 반박 (counter_thesis + 약점)

TASK:
1. 두 의견을 비교 평가하라.
2. 어느 쪽이 더 설득력 있는가? 그 이유는?
3. 최종 결정: BUY / HOLD / SKIP, 그리고 비중(0~100%).

DECISION RULES:
- 리스크 매니저의 severity >= 4 인 항목이 하나라도 있으면, 매수 비중은 50%를 넘을 수 없다.
- 애널리스트의 confidence < 0.6 이면 BUY 불가, 최대 HOLD.
- 두 에이전트가 같은 수치를 정반대로 해석한 경우, 판단 근거를 명시적으로 적어라.
- "둘 다 일리 있다"는 답변 금지. 한쪽 손을 들어줘야 한다. 단, 비중으로 강도를 조절하라.

OUTPUT FORMAT (JSON):
{
  "ticker": "...",
  "decision": "BUY|HOLD|SKIP",
  "weight": 0.XX,
  "winning_side": "analyst|risk_manager|partial",
  "reasoning": "...",            // 어떤 논거가 결정적이었는지
  "dissent_notes": "..."         // 채택 안 된 측의 핵심 우려를 한 줄로
}
```

> `dissent_notes`가 중요하다. 나중에 "왜 그때 리스크를 무시했지?"라는 사후 분석에 쓰인다.

### 2.5 Report Presenter (리포트 프레젠터) — **신규** 📊

운영진/사용자에게 보여주는 마지막 레이어. 이 에이전트는 **새로운 정보를 만들지 않는다**. Supabase에 이미 저장된 결정과 토론 로그를 읽어서 자연어로 정리만 한다.

| 항목 | 내용 |
|---|---|
| 입력 | Supabase에 저장된 trader 결정 + 두 에이전트의 원본 출력 |
| 출력 | 대시보드용 요약 텍스트, 변화 알림, FAQ형 답변 |
| 호출 빈도 | 사용자 요청 시 또는 결정 변경 시 |
| 모델 | Haiku (요약 작업, 비용 효율 우선) |

**프롬프트 골격:**
```
ROLE: 당신은 트레이더의 결정을 비전문가에게 설명하는 리포터다.
INPUT: 한 종목에 대한 전체 토론 기록(analyst, risk_manager, trader 출력).
TASK: 다음 형식으로 요약하라.
  - 한 줄 결론 (예: "삼성전자: 매수 30% 비중")
  - 왜 샀는가 (애널리스트 논거 1-2줄)
  - 무엇이 걸렸는가 (리스크 매니저 우려 1-2줄)
  - 트레이더의 판단 (어느 쪽 손을 들었는지 + 이유)

CONSTRAINTS:
- 입력에 없는 정보 추가 금지. 환각 방지가 이 에이전트의 핵심 가치다.
- 수치는 원본 그대로 인용. 반올림이나 의역 금지.
- 전문용어는 한 번에 한 줄 짧은 설명을 괄호로 부기.
```

## 3. 토론 루프 (최소 정의)

```
[Scout] ─candidates──> Supabase
                          │
                          ▼
                      [Analyst] ──buy_thesis──> Supabase
                          │
                          ▼
                   [Risk Manager] ──counter_thesis──> Supabase
                          │
                          ▼
                       [Trader] ──decision──> Supabase
                          │
                          ▼
                     [Presenter] ──summary──> Dashboard
```

**1라운드 토론을 기본**으로 한다. 즉, 애널리스트와 리스크 매니저는 각각 한 번씩만 의견을 낸다. 트레이더가 "양측 입장이 너무 갈린다"고 판단할 때만 2라운드를 트리거 (옵션, Phase 2에서 도입).

## 4. Supabase 스키마 (최소 추가분)

기존 테이블을 건드리지 않고, 새 테이블 2개만 추가한다.

```sql
-- 토론 로그 (한 ticker × 한 사이클)
create table debate_logs (
  id uuid primary key default gen_random_uuid(),
  ticker text not null,
  cycle_id uuid not null,                      -- 같은 사이클 묶기
  analyst_output jsonb,
  risk_manager_output jsonb,
  trader_decision jsonb,
  created_at timestamptz default now()
);

-- 프레젠터 캐시 (재생성 비용 절감)
create table presenter_summaries (
  cycle_id uuid primary key,
  summary_md text,
  generated_at timestamptz default now()
);
```

기존 `signals` 테이블은 그대로 두고, `signals.cycle_id`만 추가해 토론 로그와 조인할 수 있게 한다.

## 5. Strangler Fig 단계별 적용 순서

운영 중단 없이 안전하게 5각 편대로 가는 4단계.

### Phase 1 — 관찰자로 붙이기 (1~2주)
- Risk Manager를 **읽기 전용**으로 추가. 애널리스트 출력을 보고 counter_thesis만 생성해 `debate_logs`에 저장.
- Trader는 아직 기존 로직대로 작동. Risk Manager 출력은 무시.
- **목표**: Risk Manager의 출력 품질을 1~2주간 사람 눈으로 검수.

### Phase 2 — 판사 모드 전환 (1주)
- Trader 프롬프트를 "판사 모드"로 교체. 이때부터 Risk Manager 출력이 의사결정에 반영.
- 기존 Trader 로직은 shadow 모드로 병행 실행. 두 결정이 다를 때 로그.
- **목표**: 새 Trader가 기존 Trader보다 못한 결정을 내지 않는지 확인.

### Phase 3 — Presenter 도입 (3~5일)
- Presenter를 붙여 대시보드에 노출. 처음에는 내부용으로만.
- **목표**: 의사결정의 가독성 확인 및 보고용 자료 검증.

### Phase 4 — 옵션 기능 (필요 시)
- 2라운드 토론 트리거 도입.
- 거시 리스크 데이터 소스 확장 (현재는 정적 입력만).
- A/B 테스트: Risk Manager severity 임계치 조정.

---

## 6. 의사결정 필요 항목 (베이영님 확정 요청)

본격 착수 전 정해야 할 4가지.

1. **모델 매핑 확정**: 위 표의 Haiku/Sonnet 배분이 비용·품질 측면에서 맞는지.
2. **Risk Manager의 severity 임계치**: 기본값 4로 잡았는데, 실제 운영 데이터로 튜닝 필요.
3. **2라운드 토론 도입 시점**: Phase 2에서 바로 할지, Phase 4로 미룰지.
4. **Presenter 노출 범위**: 내부 대시보드만인지, 사용자 화면까지인지. (Phase 3 범위 결정에 영향.)

---

## 7. 한 줄 요약

> 기존 3각 편대를 그대로 두고, **Risk Manager(반박자)** 와 **Presenter(설명자)** 를 옆에 붙인 뒤, Trader 프롬프트를 "판사 모드"로 바꾸는 것이 이번 리팩토링의 본질이다. 코드 재작성이 아니라 **역할 재배치**다.
